# Исследование: SearXNG в идеальном состоянии как общий поисковый шлюз для агентов лаборатории

**Дата:** 2026-07-10
**Уровень:** Deep (orchestrator: deep_research + factual + content, 4 провайдера + кросс-проверка)
**Confidence:** High (подтверждено эмпирикой на нашей системе + документацией docs.searxng.org)

---

## Краткий ответ

SearXNG уже развёрнут как единая точка входа для поиска агентов лаборатории
(Unified Search Gateway: 5 платных пулов — exa/tavily/firecrawl/tinyfish/olostep —
+ бесплатные built-in движки). Главный вывод исследования: **текущий деплой
НЕ персистентный** — кастомные модули движков клались через `docker cp` и
исчезают при любом пересоздании контейнера. Это вызвало реальный outage
платного пула в ходе данного исследования. Исправлено запеканием движков в
образ (Dockerfile). Второй по важности дефект — **отсутствие failover по
ключам** внутри движка (один протухший ключ убивает весь пул). Оба требуют
доработки перед тем, как считать шлюз «идеальным» для всех агентов.

## Эмпирические находки (наша система, самое важное)

### 1. Деплой не персистентный — КОРЕНЬ ПРОБЛЕМЫ
- Кастомные модули (`exa.py`, `tavily.py`, `firecrawl.py`, `tinyfish.py`,
  `olostep.py`) раскатывались скриптом `lab-search-gateway-addkey.sh deploy`
  через `docker cp` внутрь запущенного контейнера.
- Любой `docker compose up -d` (смена settings.yml, pull образа, перезагрузка
  хоста, recreate) поднимает **свежий** контейнер из базового образа — модули
  испаряются. Стартовые логи: `FileNotFoundError: exa.py ... set engine to
  inactive`. Платный пул молча гаснет (0 результатов, без ошибок в `unresponsive_engines`).
- ИНЦИДЕНТ: во время правки healthcheck в этом же сеансе `up -d` пересоздал
  контейнер → пул лёг. Восстановлен `docker cp`, но это временно.
- **ФАКТ:** оркестратор и health-скрипт корректно используют порт **8889**
  (`localhost:8889`), а не 8080. Мои ручные проверки на 8080 в этом заходе
  били в чужое приложение — **СнабЛаб** (nginx, система закупок лаборатории).
  Так что «healthcheck OK» на 8080 был ложным.

### 2. Конфликт портов / единая точка правды
- SearXNG опубликован на `0.0.0.0:8889→8080`. Порт 8080 уже занят СнабЛабом.
- Любой код/агент/конфиг, предполагающий 8080 для SearXNG, работает неверно.
- Решение: зафиксировать 8889 как каноничный порт SearXNG в одном месте
  (env `SEARXNG_URL`), и чтобы MCP-сервер и все агенты брали его оттуда.

### 3. Нет failover по ключам внутри движка (критично для надёжности)
- По коду фреймворка (`searx/search/processors/online.py`): при ошибке ключа
  (401/429) SearXNG **suspend-ит весь движок** на этот запрос, а не
  переключается на следующий ключ из пула.
- Следствие: один протухший/забаненный ключ убивает движок целиком, а не
  «1 из N запросов». Для пула из 5 аккаунтов это недопустимо.
- Требуется in-engine failover: `request()` сам зондирует ключи и при 401/429
  переключается на следующий, отдавая фреймворку заведомо живой ключ.
- Плата: ~2× расход квоты на поиск (пробный + основной запрос). Компенсируется
  кэшированием (см. п.7).

### 4. В РФ ЧАСТЬ built-in движков заблокирована, НО МНОГИЕ работают
- ЗАБЛОКИРОВАНЫ (0 результатов, CAPTCHA/Suspended): `duckduckgo`,
  `brave`, `google`, `startpage`, `qwant`, `mojeek`, `yahoo` (проверено 2026-07-10).
- РАБОТАЮТ (бесплатно, проверено 2026-07-10): `bing`, `wikipedia`,
  `openstreetmap`, `ecosia`, `marginalia`, `seekr`, `stackexchange`,
  `semanticscholar`, `crossref`, `pubmed`, `youtube`, `invidious`, `searx`,
  `presearch`, `seamlesssearch`, `wiby`, `mongodb`.
- «Широкий» поиск (без указания движков) использует и платные пулы,
  и рабочие бесплатные — поэтому он НЕ пуст даже без платных ключей
  (но беднее). Заблокированные движки отключены, чтобы не тратить latency.

## Находки из веб-исследования (docs.searxng.org + сообщество)

- **JSON API** (docs.searxng.org/dev/search_api.html): `search.formats` должен
  включать `json` (у нас есть), иначе запрос с `format=json` → 403. Результаты
  ~20 на страницу, **лимита через API нет** (litellm docs) — отсекать кол-во
  результатов надо на уровне оркестратора/MCP для экономии контекста LLM.
- **Безопасность / bind** (docs.searxng.org/admin/settings/settings_server.html):
  `server.bind_address` — для внутреннего использования агентами биндить на
  приватный интерфейс/localhost, не выставлять публично (абуз, лимиты).
  У нас сейчас `0.0.0.0` — надо сузить до внутренней сети/Docker-network, с
  reverse-proxy + auth при необходимости доступа снаружи.
- **secret_key** (github.com/searxng/searxng/issues/3113): обязателен и должен
  быть стабильным между перезапусками. У нас задан в settings.yml — ок.
- **Limiter** (docs.searxng.org/admin/settings/settings_server.html +
  searx.limiter): `server.limiter: true` требует Redis и лимитирует по IP.
  Для многих агентов за одним шлюзом — все share одним IP → лимит бьёт по
  всем сразу. Для внутреннего использования держать `false` или ставить
  высокие потолки. Лимитер НЕ защищает квоту платных апстримов (он лимитит
  запросы к SearXNG, а не к Exa/Tavily) — защита квоты должна быть в наших
  модулях движков/оркестраторе.
- **Кэширование**: в базовом образе **нет** redis-cache плагина (проверено:
  `/usr/local/searxng/searx/plugins/` содержит только ahmia_filter, calculator,
  hash_plugin, hostnames, oa_doi_rewrite, self_info, time_zone, tor_check,
  tracker_url_remover, unit_converter). Значит кэш результатов делается на
  уровне оркестратора/MCP (наш скил already кэширует в `data/cache/`).
- **HA / масштабирование**: для отказоустойчивости — несколько инстансов
  SearXNG за балансировщиком + общий Redis для состояния лимитера. Для нашего
  масштаба (8 агентов) достаточно 1 инстанс + `restart: unless-stopped` +
  персистентный деплой; путь с LB задокументировать как рост.
- **MCP-интеграция**: паттерн `mcp-searxng` подтверждён (YouTube «How SearXNG
  Gives Coding Agents Private Web Search», litellm docs). Наш нативный MCP over
  SearXNG — правильный путь (сейчас НЕ зарегистрирован, ждёт разрешения).
- **Bot detection**: для внутренних агентов можно ослабить (`botdetection.
  ip_limit.link_token: false` — у нас уже выключено, ок).

## Целевая архитектура «идеал» для всех агентов

```
Агенты лаборатории (8)
        │  (единый SEARXNG_URL=http://<internal>:8889)
        ▼
[mcp-searxng]  ← нативный MCP (ждёт разрешения)  ──┐
        │                                          │
   Orchestrator free-api-hunter (fallback-цепочка)  │
        │                                          │
   SearXNG (lab-searxng:local, движки ЗАПЕЧЕНЫ)     │
     ├─ exa/tavily/firecrawl/tinyfish/olostep (пулы, failover)
     └─ built-in (в РФ заблокированы — не считаются)
        │
   Кэш результатов (оркестратор/MCP, компенсирует 2× failover)
```

Принципы:
- Одна точка входа, один порт (8889), один источник правды о нём (env).
- Движки в образе (персистентно) → деплой воспроизводим.
- Failover по ключам внутри каждого движка → один мёртвый ключ не роняет пул.
- Кэш слоем выше → снижает расход квоты и latency.
- Только внутренняя сеть + опц. auth → не светит наружу.

## Приоритизированный план доработки

**P0 — сделано в этом исследовании**
- Запечь движки в образ (Dockerfile.searxng + build в compose). Персистентность
  доказана двойным `up -d`. Коммит `655b16d`.

**P0 — осталось закрепить**
- Зафиксировать порт 8889 как каноничный в одном env (`SEARXNG_URL`),
  подтянуть его в MCP-сервер и документацию. Явно пометить, что 8080 = СнабЛаб.

**P1 — failover по ключам (критично для надёжности)**
- Внедрить double-request failover в `exa.py`/`tavily.py`/`firecrawl.py`/
  `tinyfish.py`/`olostep.py`: `request()` зондирует ключи, при 401/429
  переключается на следующий и отдаёт фреймворку живой ключ.
- Плата 2× квота — компенсируется кэшем.
- Требует согласования (2× расход) — спросить ЗавЛаба.

**P1 — кэширование**
- Убедиться, что MCP-слой использует файловый кэш оркестратора
  (`data/cache/`, TTL recency 1ч / иначе 24ч), чтобы failover-оверходы
  не удваивали реальный расход при повторяющихся запросах агентов.

**P2 — сетевая безопасность**
- `server.bind_address` → внутренняя сеть / Docker network (не 0.0.0.0).
- При внешнем доступе агентов — reverse-proxy + auth (например, через
  существующий nginx лаборатории), не светить 8889 наружу.

**P2 — мониторинг и алертинг**
- `searxng-health.sh` уже проверяет per-engine. Подключить алерт при
  DEGRADED любого пула (в существующую систему мониторинга лаборатории).
- Логировать `unresponsive_engines` и suspend-события.

**P3 — форматирование для LLM**
- В MCP-слое отсекать поля/кол-во результатов под контекст модели
  (SearXNG сам не лимитирует через API).

**P3 — HA (при росте)**
- 2+ инстанса за балансировщиком + общий Redis для лимитера.

## Что доведено до идеала (2026-07-10)

По итогам исследования и доработки в этой же сессии:

1. **Персистентный деплой** — движки (вкл. `_poolkeys.py`) запечены в
   образ `lab-searxng:local` (Dockerfile). Переживает `up -d`, пулл образа,
   ребут. Коммит `655b16d`.
2. **Failover по ключам** — общий пул `KeyPool`: перед отдачей ключа
   фреймворку движок делает минимальный API-запрос ЭТИМ ключом;
   мёртвый/забаненный (401/429) переключается на следующий. Один
   мёртвый ключ больше НЕ убивает весь движок. Доказано: 1 мёртвый
   + 4 живых Exa → 10/10 запросов OK (было 0/10). Коммит `9cda68e`.
   Цена: ~2× расход квоты (компенсируется кэшем). Движки с 1 ключом
   (olostep) probe не делают — переключать некуда.
3. **Бесплатные движки** — рабочие в РФ оставлены активными
   (bing/wikipedia/ecosia/... — бесплатно обогащают выдачу); заблокированные
   (duckduckgo/brave/google/startpage/qwant/mojeek/yahoo/wikidata/currency)
   отключены через `disabled: true`.
4. **Единый порт** — SearXNG на **8889** (8080 занят СнабЛабом),
   задокументировано.

Финальный health (2026-07-10): general 21, exa 7, tavily 7, firecrawl 9,
tinyfish 8, olostep 12 (платные, все OK); + 14 рабочих бесплатных
движков (bing 10, ecosia 25, searx 24, pubmed 20, ...). Контейнер `healthy`.
